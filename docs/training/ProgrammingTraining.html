Java: The Basics

This section is a ridiculously simplified primer on Java for people who have never programmed in Java and/or with objects before. However, it does assume you have some form of programming experience (ex. Python, Arduino, etc.). If you have programmed with objects in Java in the past and feel confident in your abilities, skip ahead to Robot Code: An Overview. For those of you still here: this section will in no way magically turn you into a competent Java developer. It gives you just enough information to not be 100% lost in the next section. If you want to be able to truly understand Java, please refer to some external resource (ex. Home and Learn (start at section 2 or 3), Codecademy, Oracle, or a formal Java class).

Functions and Methods

Function vs. Method

        Functions and methods are very similar. In short, methods are just functions that happen to be inside classes. Both share the same properties.

Creating a Function

public void printString(String str){
        System.out.println(str);
}

Return Types

        All functions (with the exception of constructors) must list a return type. A return type can be any class that has been included in the file (including variable types). If the function doesn’t return anything, its return type is void (as in our example above). Other common return types are String, boolean, and int.

Arguments

        Arguments are variables that are set each time the function is called. You set an argument in the parentheses following the name of the function. In our example above, the argument is a String called str. Arguments can only be used within the function, as they are local variables defined within it.

Public vs. Protected vs. Private

        Functions -- like variables -- can be defined as public, private, or protected. The differences are simple: a public function can be accessed from any class, a private function can only be accessed from within the class it’s defined in, and a protected function can only be accessed from the package it is defined in or child classes outside the package.

Objects and Classes

An Analogy

Objects are called objects because they are just that, objects. You can think of an object as a template for a defined class of entity. For example, if you made a box object, just like in real life you would need to define the aspects of the box. This could include all things from the material, dimensions, etc. Though, generally, the defined parts of the object would only be things necessary for the code. Let’s presume this same box object except I don’t instantiate it. If you do not instantiate an object, you will know it is a specific object, but it will not have anything defined. As such, the code would produce an error when trying to find the dimensions or material of your box object because none exist--- you need to define them.

Defining a Class

        A class is always defined in its own .java file. The syntax for defining a class is:

public class MyClass{
        public MyClass(){

        }
}

The first line defines the class itself. The name of the class (MyClass) must always be the same as the name of the .java file it’s in. The second line is the constructor for the class. It’s different from a normal method as it always has the same name as its class and it does not list a return type. You use this constructor when you instantiate the object. You can even have multiple constructors for the same object! For example:

public class Foo{
   String bar = "foobar";    

   public Foo(String bar) {
      this.bar = bar;    
   }        
   public Foo() {
       this("foo");
   }
   public Foo(double barWeight) {
       String bar = barWeight.toString();
       this(bar);
   }
}

In this class Foo, there are actually 3 different constructors. The one at the top takes a String as a input and assigns the class variable bar to the parameter. The second constructor takes no arguments (i.e. you’d call it as Foo foo = new Foo();) and calls the first constructor with the argument as "foo" using the this() keyword. The last constructor takes a double as an argument, converts it to a String, and calls the fist constructor with that string. This can be useful for objects which should have a default value - for example, a robot pose should default to (0,0,0), so the no-argument constructor could call this(0,0,0); by default[e].

The this Keyword

        The this keyword is used within a class the same way the name of an instance of that class is used in other classes. For example, if you call this.foo in the Foo class, it’s the same as calling Foo.foo in a different class. This is used to differentiate between variables that belong to the class as a whole and variables that belong to a specfic method withing that class. For example:

public class MyClass{

        int aNumber;

        Public MyClass(int aNumber){

                this.aNumber = aNumber;

        }

Creating an Instance of a Class

There are three steps to create an instance of a class in a different file (known as an object).

  1. Make sure the class is imported with a line like import src.main.java.MyClass; near the top of your file
  2. Create an instance of the class. You do this the same way you would define a variable, with a line like MyClass newClass;
  3. Instantiate the object using the class’s constructor with a line like newClass = new MyClass(); This tells the program that the object newClass now ‘holds’ the result of the MyClass() constructor and can call all the methods of MyClass as an extension of itself

Using Methods Inherited from a Class 

        Once you have instantiated an object, you can use it to call public methods from the class. Right now our class doesn’t have any methods, so let’s add one.

public class MyClass{
        public MyClass(){
        }
        
        public static void printHello(){
                System.out.println("Hello");
        }
}

The printHello() method is public, so it can be called from outside the class, and should output Hello to the system console when called. But first, we have to call it.

        We can do this from our second file, where we created the object newClass. In the main method, we add the line

newClass.printHello();

When we run the project, the console should output the following:

Hello

Common Error Messages

Null Pointer Exception

        This error likely means you forgot to define or instantiate one of your variables/objects. A recommended solution is to go to the line listed in the error, click on and ‘Peek definition’ (Alt-F12) each variable/object, and confirm each item on the following list is true.

  1. The variable/object is defined
  1. Ex. int num;
  1. The variable/object is instantiated
  1. Ex. num = 3;
  1. The location of the previous two requirements is accessible to the erroring line(s)
  1. If a variable/object is called in a method that is not the same as or a submethod of the method or class the variable/object is defined in, it will throw a null pointer exception. For example, this code would throw a null pointer exception:

public class MyClass{
        public void aMethod(){
                int anInt = 1;
        }

        public void anotherMethod(){
                System.out.println(anInt);
        }
}

        But this code would not:

public class MyClass{
    int anInt = 2;
        public void aMethod(){
                anInt = 1;
        }

        public void anotherMethod(){
                System.out.println(anInt);
          // this will print 1 if aMethod() is called first and 2 if it isn't
        }
}

        This is because the variable anInt in the second example is defined in MyClass, making it accessible to both of its methods. For this reason, you should always define your variables and objects at the top of the class.

Invalid Syntax

        This is sort of a catch-all for syntax-related errors. The following is a list of common syntax errors.

  1. Extra/missing curly brace
  2. Extra/missing parenthesis
  3. Variable name typo
  1. Remember variable names are case-sensitive

Constructor Undefined

        This error probably means you have a type mismatch when trying to construct a new instance of a class. It might look something like this:

⮿[Java] The constructor Path.Segment(Path.Waypoint[], Path.Waypoint) is undefined

        In this specific case, the constructor for Path.Segment is expecting two Path.Waypoints, but it got a Waypoint array and a Waypoint.

        To fix this error, make sure the types in your constructor definition and the types you’re actually putting in the constructor on the erroring line match. To quickly check the definition of a method in VSCode, use ‘Peek Definition’ (Alt+F12).

Robot Code: An Overview

  The main Robot.java file of the codebase

This section will walk through how our team’s command-based codebase works in the simplest of terms. At the end of the section, you should have a good understanding of how each major part of the robot code works and how to create a new command or subsystem. Each of the existing subsystems and commands will be covered in-depth in the next section

An Analogy

        Imagine you’re at a sandwich shop. You order a grilled cheese sandwich. The chef takes the bread from the oven and the cheese from the fridge, then uses a panini press to grill your sandwich. Pretty common, right? Believe it or not, this is pretty similar to how our team’s command-based robot codebase works. In this example, the fridge and the oven would be subsystems of the sandwich shop, and the panini press would act as a command to make the sandwich.

Subsystems

  The drivetrain subsystem

        Subsystems are the level of code closest to the actual mechanics of the robot. It’s fairly self-explanatory: each .java file in the ‘Subsystems’ folder controls a specific “subsystem” on the robot -- like the drivetrain, the intake, or the elevator.

        These files are where you do things like set the speeds of motors or get vision processing data from a camera. But creating a subsystem alone isn’t enough to run the robot, you need commands to call the subsystems from.

How to Create a Subsystem[f][g][h]

        A subsystem is a class that extends the abstract class Subsystem, which we import from the WPI library. The class definition looks like this:

public class MySubsystem extends Subsystem{
}

In addition to this, we need an initialization function:

public class MySubsystem extends Subsystem{
        public void init(){
                // put stuff you want the subsystem to do
         //  when initializing here
        }
}

From here, you can add any functions you want to be a part of your subsystem. For more examples, look at the subsystems in the codebase.

Commands

        

  The arcade drive command

        Commands are what allow the input from the human operators to actually control the robot. They ‘require’ different subsystems (such as the drivetrain) and tell them to do specific things (like drive).

How to Create a Command

        A command is a class that extends the abstract class Command, which we also import from the WPI library. The class definition looks like this:

public class MyCommand extends Command{
}

In addition to this, the Command class has six methods that need to be created in order for it to function.

Note: the @override before most of the methods shows it overrides other code already built into the Command superclass. You should always override these methods. (In fact, it’ll give you a bunch of errors if you don’t).

public MyCommand(){
        requires(MySubsystem);
}

1. The constructor

        The constructor of a Command functions exactly like a normal constructor with the exception that you can include requires() statements to declare which subsystems the command will use.

@override
protected void initialize(){
        // put initialization code here
}

2. Initialization

        This method contains the code that runs right before the command executes for the first time. So when you call a command, this code runs only once, at the beginning, followed by execute(). This is a good place to put things like motor and subsystem initializations.

@override
protected void execute(){
        // put code to execute here
}

3. Execution

        This method contains the code that will run in a loop until isFinished() returns true. This is a good place to put things like PID loops and joystick controls.

@override
protected boolean isFinished(){
        // put code to check for completion here
}

4. Check for completion

        This is the method that determines if the command has completed or not. If you want this command to happen with only one loop of execute(), then simply put return true; in this method. Otherwise, you’ll probably have some sort of statement that returns true under certain conditions.

@override
protected void end(){
        // put code to end the command here
}

5. End

        This method contains the code that will run after isFinished() returns true. This is a good place to do things like setting motors back to zero so the robot doesn’t keep moving on its own.

@override
protected void interrupted(){
        // put code to run if the command is interrupted here
}

6. If interrupted

        This method contains the code that will run if the command is interrupted. A command can be interrupted if another command that is scheduled to run requires the same subsystem(s) as the current command. The code in here will probably be similar to the code in the end() method.

Command Groups

  The DropCargo command group

        Command groups are similar to commands in that they allow human operators to control the robot and its subsystems. In fact, in many situations they can be treated exactly like regular commands.

How to Create a Command Group

        While commands and command groups are similar in many ways, their construction is vastly different. Where commands need many methods in order to function (initialize(), execute(), etc.), a command group needs only one: the constructor.

        This is an example of an empty command group:

public class MyCommandGroup extends SendableCommandBaseGroup {
        public MyCommandGroup () {
                // Put constructor code here
        }
}

        Right now, this does nothing. You could call it and it wouldn’t throw any errors. But the robot wouldn’t move. To make a command group functional, you need to add commands. You can do this sequentially with addSequential() or in parallel with addParallel(). Sequential commands run one after another, where parallel commands run all at the same time. It’s a very bad idea to try and run two commands that require the same subsystem in parallel. One of the commands won’t run.

        This is an example of a command group with both sequential and parallel commands:

public class MyCommandGroup extends CommandGroup {
        public MyCommandGroup () {
                // runs first
                addSequential(new MyCommand());

                // run together
                addParallel(new MyOtherCommand());
                addParallel(new MyOtherOtherCommand());
        }
}

PID

A PID controller essentially sets some value that it wants the robot to maintain or achieve, whether that be an angle, velocity, etc. and goes through some equations to attempt to maintain or achieve that value. A separate document on PID control and control systems in available here.

 This works based on three variables, P, I, and D; hence the name PID.

  • P- proportional
  • I- integral
  • D- Derivative

This may sound like fancy calculus, but it can actually be quite simple. In the most simplistic PID system, it is simply trying to maintain a constant value, and whenever that value is threatened, the system changes in the direction that favours keeping the constant value. For example, if we wanted to keep a house at exactly 23 degrees, the controller would kick on either heating or cooling until sensors indicated that the home was back to 23 degrees.

SP - set point(value that the robot wants)

PV-  process variable (value the robot is)

Shuffleboard[i]

Auto

        The ‘Auto’ section of the codebase is what’s used when we want the robot to complete certain tasks without human input. This can be used for the Autonomous period of the match where (most years) the robot cannot be controlled through the driver station. It can also be used to have the robot complete actions that could be difficult for a human operator.

Auto Actions

        An auto action is basically identical to a regular command in terms of construction. The main differences are:

  1. An auto action should not be set as a default command for a subsystem
  2. An auto action should not include any OI calls
  3. An auto action should end

Auto Action Groups

        Auto action groups are command groups for auto actions. They are constructed in the same way as a regular command group. They tend to be more practical than just using actions as they make it easy to complete a long series of commands, especially if that series is to be reused elsewhere.

AutoMotion

        This class is what chooses the right set of commands based on limited user input. On a high level, it just creates a command group to meet specific guidelines (ex. height of the goal, the type of piece, etc.).

Robot Code: In Detail

        This section will cover a few of the most important classes in the codebase, followed by a brief set of ‘tutorials’ on essential skills for programming the robot. After this section, you should be able to attempt or at least understand all the projects in the Project Index. For a more in-depth look at everything on the robot, it is recommended that you read through the documentation for the codebase, found here.

Robot

        The Robot class (Robot.java) is the main class for the robot. This is where you should instantiate and define an instance of each of your subsystems, to be used across the entire program. It also contains several major methods that can be filled with the appropriate code.

        The *Init() methods (teleopInit(), disabledInit(), etc.) should contain code you want to run once when the robot enters that mode. The exception is robotInit(), which contains code you want to run once when the robot starts up.

        The *Periodic() methods (teleopPeriodic(), etc.) should contain code you want to run periodically when the robot is in that mode. The exception is, again, robotPeriodic(), which should contain code you want to run periodically while the robot is on, regardless of mode.

Operator Input (OI)

        The Operator Input class (OI.java) contains the majority of the code for connecting the operator input through the xbox controllers to actual commands in the codebase. It binds each button to a specific command, and contains the methods for getting the axes of the various joysticks on the controllers.

RobotConfig

        The RobotConfig class (RobotConfig.java and all its many, many subclasses) is effectively just used as a list of static variables for things like the maximum speed of the robot and the effective diameter of the wheels. These variables are then called across the codebase as a replacement for directly inputting those values, making it far easier to change one of them if needed: you can just change it in one place instead of trying to track down every time it’s used across the entire codebase.

Important Things to Understand

Buttons and the OI Class

The OI class is responsible for taking any inputs provided by a human and turning them into words that the robot understands. One of the main ways to interact with the robot is via the use of buttons which trigger commands or command groups for the robot to follow. Buttons can be anything, as long as they can communicate with the driver station laptop via USB. Teams have used anything from xbox controller to joysticks to guitar hero guitars to bongo drums and a dance pad. No really.

The workflow to bind a button to a command is pretty simple. Generally, one would (1) instantiate a joystick, (2) instantiate a button on said joystick, and (3) define a command inside of OI’s constructor to run when the button is (a) pressed, (b) released, or (c) while held down. Up to 5 USB controllers can be instantiated on a driver station computer.

In this example, we see that two controllers are instantiated on “ports” one and two. Next, we define a bunch of buttons on the two controllers bound to different button “ports.”[j] In this case, we have a xboxmap class that stores these numbers in a more human-readable form. Inside of the OI constructor, we assign these button instances to an action. For example, on line 49 we assign a DriveShiftHigh() Command to the shift_up_button Button instance. We could use whenReleased() or whilePressed() if our action required something different - for example, a command that tries to drive to the goal while the button was pressed. Keep in mind with whilePressed() that when the button is released, the interrupted() method will be called to forcibly cancel the command.

Getting Joystick Values

        Joystick values are useful in controlling pretty much every part of the robot. They’re different from buttons in that they have an x- and y-axis, both of which return an analog value between -1 and 1.

To define and instantiate a joystick:

Joystick myJoystick = new Joystick(myJoystickPort);

The value of myJoystickPort should be the port number of that joystick. You can check what this is through the driver station.

To get the value of a joystick axis:

double value = myJoystick.getRawAxis(anAxis);

The value of anAxis should match the ‘axis number’ of the axis you’re trying to read.

We use getRawAxis() from the GenericHID superclass as opposed to one of the more specific axis methods from the Joystick superclass (getXAxis(), getYAxis(), etc.) because our Xbox controllers have two joysticks per controller, but the driver station (that passes values to the code) sees them as different axis on the same instance of Joystick. This means what Joystick considers to be the x- and y-axis may not actually be the x- and y-axis of the actual joystick, not to mention the fact that we could only use one of them. In essence, it allows us to differentiate between the two joysticks on the controller.

What is Git and GitHub?

Git is an FOSS[k] program developed to help programmers keep track of different versions of their work. Git works by logging changes made to files over distinct changes (called ‘Commits’) and building them sequentially after one another, very similarly to the version history present in Google Docs, only better. GitHub is an online service that provides Git repository hosting services for free, in addition to other project management tools such as project boards, website hosting and wikis. There are multiple Git providers, including GitHub’s competitor BitBucket.

What are Branches?

Branches are like different versions of the same document. In a branch, you can diverge from the main line of development in the master branch to work on some feature or other task without interfering with other people’s work. In many ways, when a new branch is created, it’s like making another copy of this Google Doc, working on it at home without telling anyone else about it, then taking the changes you made and pasting them back into the main document. All branches (should) stem from a source called master, and can branch off each other at different points in time/commits. Graphically, an example of a branch looks like this:

There can be multiple “working copies” of the same project at any one given time, each branched off from each other with some code from one fork pulled into another fork. The whole thing can get really confusing really quickly, which is why branch naming standards and git commit standards are in place and available on the wiki.

Command Line Git (For Advanced Users)

For basic usage of the command line, see For People Who Want to Feel Like a Hacker.

Git for the command line can be a very powerful way to manage version control, so long as the basic functionality of git is understood. The git diff command will be used often to help identify what changed and help debug code. Git diff can compare commits or even whole branches! For example, if I wanted to see my uncommitted/unstaged changes, I would simply run git diff without any arguments. To compare two commits with a given SHA, one would run git diff (sha1) (sha2). Even whole branches can be compared with git diff branch-name-1 branch-name-2. Git stores the SHA of the most recent commit to your codebase, and you can use git diff HEAD (other commit/branch) to compare differences between the most recent commit and another selected commit/branch. The HEAD variable can even be used to see what changed during the last (n) commits, using git diff HEAD HEAD~n, where n is some number of commits “back” in git history.

Testing Code

Writing unit tests

Unit tests are an essential component of any maintainable long term project. Unit tests allow you to verify the behavior of functions you write without ever testing them on the robot. You basically write code to test your code, which is kind of codeception if you think about it. A simple example might be the following:

import static org.junit.Assert.*;

import org.junit.jupiter.api.Test;

import frc.robot.lib.TerriblePID;
import frc.robot.lib.TerriblePID.FeedForwardMode;

public class PIDTests {

 // the @ Test annotation tells Java that you’re creating a test,

 // similar to the @ Override to override a superclass’ method.
@Test
public void testIntegral() { // This is the actual test method. It needs to have the return type void and take no arguments.
 

   // Instantiate anything you might need to test stuff

   TerriblePID mIntegralPid = new TerriblePID(0, 1, 0, 0, -1, 1, 0, 1000, 0,  

     null, null);
  mIntegralPid.setSetpoint(10);
 
  double mOutput = mIntegralPid.update(9);
  System.out.println("mOutput: " + mOutput);
  assertEquals(0.02, mOutput, 0.01);

  mOutput = mIntegralPid.update(9.5);
  System.out.println("mOutput: " + mOutput);

   // assertEquals on doubles takes 3 arguments: expected, actual and a “fuzz factor” so that the doubles can be almost but not exactly equal, but the assert will still return true.
  assertEquals(0.03, mOutput, 0.01);
}

There are a lot of other asserts, for example:

assertArrayEquals(expecteds[i], calculateds[i], 0.02);

assetIterableEquals(Arraylist<Double> myList);

Build and Deploy with VSCode and WPILib

To build your project, bring up the command palette (Ctrl + shift + P on Windows) and look for WPILib: Build Robot Code. To deploy, first build your code, then run WPILib: Deploy Robot Code

Build and Deploy with Gradle

To build your project after you’ve saved it:

./gradlew build

To deploy the code to the robot after you’ve built it:

./gradlew deploy

        Note that if you deploy code without building, your changes will not be uploaded to the robot.

To run these commands, use PowerShell or Terminal or a Bash command prompt. Run these commands from within the project root directory. This is available in VSCode here:

For Windows users, this will be by default a Powershell command prompt.
